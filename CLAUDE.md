# CLAUDE.md

個人用ツールを管理するリポジトリ。

## 構成

- リポジトリ直下 … 各ツール。ツールごとに専用ディレクトリを持つ（`tools/` のような親でまとめない）。
- `docs/` … ドキュメント（設計メモ・リファレンス）。
- `_/` … サンプル・試作の雑多な置き場（コード片〜サンプルツール全体）。

## 規約

- ツールごとにリポジトリ直下でディレクトリを分け、各ツールに `README.md` を置く。
- 後から自分が読み返したときに役立つレベル（目的・前提・非自明な挙動）を残す。
- 変数名・引数名は直感的にする。意味の広い汎用語（`value`・`data` など）は避け、役割が伝わる名前を付ける。
- コードは基本的に Node.js。追加依存は極力避ける。依存が必要なツールは**自ディレクトリに `package.json` を同梱**して閉じる
- 関数・クラスには **JSDoc を必須** とする（`@param` / `@returns`、必要なら `@throws`・使用例）。引数/戻り値はここで示す。
- テスト: `node --test "*/*.test.js"`（Node 標準ランナー）。リポジトリ直下のツールディレクトリ配下のテストを一括実行する。

## テストの書き方

主要 OSS の流儀に倣い、**テストコード自体を仕様として読める**ことを目標とする。

- **テスト名に仕様を込める**: 第1引数のテスト名に「入力条件 → 期待結果（What）」を1文で書く。
  関数単位のテストは `関数名: 振る舞い` の形にする（例: `markDataLine: 非一致なら末尾に空セルを付けて列数を揃える`）。
- **本体は AAA 構造**: 「準備 → 実行 → 検証(assert)」の順に並べ、空行で区切る。
- **コメントは Why のみ**: テスト名で分かる What は書かない。
  非自明な理由（回帰防止の背景、エッジケースの根拠、トリッキーな前提）だけを短く残す。
- 複数 assert がある場合は、どれが落ちたか分かるよう assert にメッセージを添えてよい。

## CLI の規約

CLI ツールはスイートとして一貫性を保つ。

- **引数解析**: `node:util` の `parseArgs` を使う（生 `process.argv` 参照は避ける）。
  `--flag=value`・短縮形・未知フラグ検出が揃い、フラグ追加にも耐える。
- **位置引数 vs フラグ**: 「**何を**処理するか＝位置引数」「**どう**処理するか＝フラグ」を基本とする。
  詳細な判断表・指定方法の優先順位は [docs/cli-design.md](docs/cli-design.md) を参照。
- **標準入力**: 標準出力へ書き出すフィルタ／変換系は、主入力をファイル or 標準入力で受ける。
  入力パスに `-`（または省略可能なら省略）を指定したら標準入力から読む。
- **出力の方針**: stdout / stderr / `--out` / `-q` の使い分けは下の「出力の既定方針」に従う。
- **ヘルプ**: `-h` / `--help` で使い方を標準出力へ出して正常終了（exit 0）。
  引数不足・不正時は使い方を標準エラーへ出して `exit 1`。
- **使い方文言**: `USAGE` 定数にまとめ、ヘルプとエラー時で共用する。

### 出力の既定方針（stdout / stderr / --out / -q）

新しいスクリプトは、特段の理由がない限り次の入出力契約を**既定**として作る。
目的は「データはデータ専用ストリームに、メタ情報は別に」分け、**パイプ合成とトークン節約**を
同時に満たすこと。

| 要素    | 既定の役割                                                                    |
| ------- | ----------------------------------------------------------------------------- |
| stdout  | 処理結果データ**のみ**。パイプで次段へ流せる純粋なデータに保つ                |
| stderr  | 完了サマリ・進捗・エラー文言。stdout には混ぜない                             |
| `--out` | `-o` / `--out <file>` 指定時はデータをファイルへ書き、**stdout は沈黙**させる |
| `-q`    | `-q` / `--quiet` 指定時は stderr の**サマリのみ**抑制（エラー文言は残す）     |

- **データ出力（stdout ↔ `--out` は排他）**: 出力経路を2つ用意する。**標準出力**はパイプで次段へ繋ぐ用、
  **`--out <file>`** はツール内で直接ファイルへ書き出す用。既定は stdout、`--out` 指定時はファイルのみで、
  両方には出さない。`--out` を使えばエージェントが大量データを文脈に取り込まずに済む（トークン節約）。
  途中段はパイプで stdout を流し、最終段だけ `--out` でファイル化する。
- **サマリは stderr に1行（既定 ON）**: `<ツール名>: 読込 N 行 → 出力 M 行（…）→ <出力先>` の形に揃え、
  `grep -c` / `dd` / `git` 流の「件数」を基本にする。エージェント経由では stdout/stderr はまとめて
  捕捉されトークンになるため、サマリは簡潔な1行に留める。
- **エラーは常に stderr + `exit 1`**: `-q` でも抑制しない（黙らせるのは成功時のサマリだけ）。
- **このパターンが過剰なツール**: 主出力が stdout データでないもの（ファイル群生成・DB 投入など）は
  `--out` を持たず、全出力（サマリ含む）を stderr に出す（例: `tsv-to-properties` / `excel-to-oracle`）。
  この場合も「データの出力先と無関係に、メタ情報は stderr・`-q` で抑制可」は共通とする。
  **入力側も同様**で、こうしたツールは主入力をファイル指定のみで受ければよい（stdin 対応は不要）。
  stdin/ファイルの両対応が必須なのは「標準出力へ書き出すフィルタ／変換系」だけ（上記「標準入力」参照）。

> シェル側のリダイレクト・パイプ合成の挙動（`>`・`2>&1`・`--out` との合成など）は
> [docs/shell-redirect.md](docs/shell-redirect.md) にメモとして分離した。

## Git 運用

- 本リポジトリは小規模なツール（直下の各ツールディレクトリ）と、将来追加し得る Claude Code 資産（ `skills/`・`agents/` ）を扱うだけなので、**`main` に直接コミットしてよい**（機能ブランチ必須としない）。
- コミットは引き続き1つの論理変更単位でまとめる。
