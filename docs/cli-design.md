# CLI 設計メモ（位置引数 vs フラグ・指定方法の優先順位）

自分用リファレンス。CLI ツールを作るときの設計判断の材料。
このリポジトリで守る実ルール（`parseArgs`・ヘルプ・終了コード・出力の既定方針など）は

## 位置引数 vs フラグ

「**何を**処理するか＝位置引数」「**どう**処理するか＝フラグ」を基本に、下表で判断する。
フラグには短縮形（`-c` 等）も用意する。

| 観点             | 位置引数にする                     | フラグにする                     |
| ---------------- | ---------------------------------- | -------------------------------- |
| 必須 / 任意      | 必須の主オペランド                 | 任意・省略可能                   |
| 妥当なデフォルト | ない（必ず要る）                   | ある（省略時に既定値）           |
| 真偽値か         | ×（位置だと意味不明）              | ◯ boolean は必ずフラグ           |
| 個数             | 2〜3個まで（順序を覚えられる範囲） | 多数のオプション                 |
| 意味の自明さ     | 入力・出力など自明なもの           | 挙動の微調整・モード切替         |
| 将来の拡張       | 増えない前提のもの                 | 増えそうなもの（順序が壊れない） |
| 可変長           | 末尾に可変個（複数ファイル）はOK   | 複数値は繰り返しフラグで         |

経験則: 位置引数が4つ以上になったら設計を疑う。boolean とデフォルト持ちは原則フラグ。

## 指定方法の優先順位

設定が多ければ設定ファイル、機密・環境差は環境変数、機能が増えたらサブコマンドを検討する。
優先順位は **デフォルト < 設定ファイル < 環境変数 < CLI フラグ**。
小さい単機能ツールは「フラグ + 位置引数 + 標準入力」で十分（過剰な仕組みは入れない）。
